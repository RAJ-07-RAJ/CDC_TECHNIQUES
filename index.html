01

Two Flip-Flop (2FF) Synchronizer

FUNDAMENTAL

// Block Diagram

// Use Case

  • Single-bit control signals (enable, reset, interrupt, flag)
  • Level signals that stay HIGH for multiple destination clock cycles
  • Low-latency CDC where MTBF is acceptable

// Key Insight

The source signal must remain stable for at least 2 destination clock cycles. If the pulse is shorter than one destination clock period, the destination may miss it entirely. The 2FF chain gives the metastable first flip-flop a full clock period to resolve before being sampled by FF2.

// Pros

  • Simplest and lowest-area CDC solution
  • Only 2 clock cycles of synchronization latency
  • Universally supported by all synthesis tools
  • Well understood MTBF characteristics

// Cons

  • Fails if source signal is a short pulse (pulse can be missed)
  • Cannot be used for multi-bit buses directly
  • Non-zero MTBF — failure probability exists
  • No feedback to source domain

// Verification Checklist

  • Verify source signal stays HIGH ≥ 2× destination clock period in all corner cases
  • Check MTBF calculation meets system reliability target
  • Confirm no combinational logic between FF1 and FF2 (CDC lint rule)
  • Simulate with asynchronous clock relationship (non-integer ratio)
  • Run CDC lint (SpyGlass/Questa CDC) to flag unsynchronized crossings
  • Check reset synchronization — both FFs must reset cleanly
sync_2ff.v / tb_sync_2ff.v 
module sync_2ff (
    input  wire clk_b,
    input  wire rst_n,
    input  wire async_in,
    output reg  sync_out
);
reg sync_ff1;

always @(posedge clk_b or negedge rst_n)
begin
    if(!rst_n)
    begin
        sync_ff1 <= 0;
        sync_out <= 0;
    end
    else
    begin
        sync_ff1 <= async_in;
        sync_out <= sync_ff1;
    end
end
endmodule
`timescale 1ns/1ps
module tb_sync_2ff;
reg  clk_a, clk_b, rst_n, async_in;
wire sync_out;

sync_2ff dut (
    .clk_b(clk_b), .rst_n(rst_n),
    .async_in(async_in), .sync_out(sync_out)
);

// clkA = 10ns, clkB = 16ns
initial begin clk_a = 0; forever #5  clk_a = ~clk_a; end
initial begin clk_b = 0; forever #8  clk_b = ~clk_b; end

initial begin
    rst_n = 0; async_in = 0;
    #20; rst_n = 1;
    @(posedge clk_a); async_in = 1;
    repeat(2) @(posedge clk_a); async_in = 0;
    #200; $finish;
end

initial $monitor("TIME=%0t | ASYNC_IN=%b | SYNC_OUT=%b",
    $time, async_in, sync_out);
endmodule
02

Three Flip-Flop (3FF) Synchronizer

HIGH RELIABILITY

// Block Diagram

// Use Case

  • Same as 2FF but where higher MTBF is required
  • High-speed interfaces where metastability window is tight
  • Safety-critical applications with strict reliability requirements

// Key Insight

The extra flip-flop gives FF1 a full extra clock cycle to resolve metastability before FF2 samples it. MTBF improves exponentially — typically by 10⁶× or more. This matters at high clock frequencies where the metastability resolution window shrinks.

// Pros

  • Significantly higher MTBF than 2FF
  • Still very simple — just one extra flip-flop
  • Preferred for high-frequency designs
  • No change in interface protocol

// Cons

  • 3 clock cycle latency vs 2 for 2FF
  • Same limitation: cannot handle narrow pulses
  • Same limitation: not for multi-bit buses
  • Slightly more area (one extra FF per bit)

// Verification Checklist

  • Same checks as 2FF, plus verify MTBF improvement meets target (calculate using technology library parameters)
  • Confirm synthesis tool does not merge FF1→FF2→FF3 chain (use DONT_TOUCH or synthesis attributes)
  • Ensure CDC tool recognizes the 3FF chain as a valid synchronizer structure
  • Verify reset propagation: all 3 FFs must reset to known state simultaneously
  • Check timing report: FF1→FF2 path must be multi-cycle false path (source clock, not dest clock)
sync_3ff.v / tb_sync_3ff.v 
module sync_3ff (
    input  wire clk_b,
    input  wire rst_n,
    input  wire async_in,
    output reg  sync_out
);
reg sync_ff1;
reg sync_ff2;

always @(posedge clk_b or negedge rst_n)
begin
    if(!rst_n)
    begin
        sync_ff1 <= 0;
        sync_ff2 <= 0;
        sync_out <= 0;
    end
    else
    begin
        sync_ff1 <= async_in;
        sync_ff2 <= sync_ff1;
        sync_out <= sync_ff2;
    end
end
endmodule
`timescale 1ns/1ps
module tb_sync_3ff;
reg  clk_a, clk_b, rst_n, async_in;
wire sync_out;

sync_3ff dut (
    .clk_b(clk_b), .rst_n(rst_n),
    .async_in(async_in), .sync_out(sync_out)
);

initial begin clk_a = 0; forever #5 clk_a = ~clk_a; end
initial begin clk_b = 0; forever #8 clk_b = ~clk_b; end

initial begin
    rst_n = 0; async_in = 0; #20; rst_n = 1;
    repeat(5) begin
        @(posedge clk_a); async_in = 1;
        repeat(2) @(posedge clk_a); async_in = 0;
        #40;
    end
    #200; $finish;
end

initial $monitor("TIME=%0t | ASYNC_IN=%b | SYNC_OUT=%b",
    $time, async_in, sync_out);
endmodule
03

Toggle Synchronizer

PULSE SAFE

// Block Diagram

// Use Case

  • Transferring short pulses or events between clock domains
  • Event notification where pulse width < destination clock period
  • Slow-to-fast AND fast-to-slow CDC pulse transfer

// Key Insight

Instead of sending a level signal, the source toggles a bit on every event. The destination 2FF synchronizer captures the toggle, and an XOR edge detector generates a pulse. Even a very narrow source pulse is remembered as a level change — it cannot be lost.

// Pros

  • Short pulses can never be lost — toggle persists until captured
  • Works for both slow-to-fast and fast-to-slow domains
  • Simple logic: one toggle FF at source + 2FF sync + XOR at dest
  • No handshake overhead for pulse transfer

// Cons

  • Source must NOT send a second pulse before the first is captured
  • XOR edge detection adds one extra destination clock of latency
  • Cannot transfer data payload — only the event/pulse
  • Requires 3 destination FFs (B1, B2, B3) for edge detection

// Verification Checklist

  • Verify back-to-back pulses at source are separated by ≥ 2 destination clock periods
  • Confirm pulse_out is exactly 1 destination clock cycle wide
  • Check XOR logic: pulse_out = b2 ^ b3 (not b1 ^ b2 — race risk)
  • Simulate at minimum and maximum clock ratio extremes
  • Verify toggle resets to known state after reset release
  • CDC lint: flag the toggle signal as intentional CDC crossing (false path on data)
toggle_synchronizer.v / tb_toggle_synchronizer.v 
// ── SOURCE DOMAIN ──
module toggle_src (
    input  wire clkA, rst, pulse_in,
    output reg  toggle
);
always @(posedge clkA or posedge rst) begin
    if (rst)  toggle <= 1'b0;
    else if (pulse_in)
        toggle <= ~toggle; // toggle on every event
end
endmodule

// ── DESTINATION DOMAIN ──
module toggle_dest (
    input  wire clkB, rst, toggle_async,
    output wire pulse_out
);
reg b1, b2, b3;
always @(posedge clkB or posedge rst) begin
    if (rst) {b1,b2,b3} <= 3'b0;
    else  begin
        b1 <= toggle_async;
        b2 <= b1;
        b3 <= b2;
    end
end
assign pulse_out = b2 ^ b3; // XOR edge detect
endmodule

// ── TOP ──
module toggle_synchronizer (
    input  wire clkA, clkB, rst, pulse_in,
    output wire pulse_out
);
wire toggle_sig;
toggle_src  u_src  (.clkA(clkA), .rst(rst), .pulse_in(pulse_in), .toggle(toggle_sig));
toggle_dest u_dest (.clkB(clkB), .rst(rst), .toggle_async(toggle_sig), .pulse_out(pulse_out));
endmodule
`timescale 1ns/1ps
module tb_toggle_synchronizer;
reg  clkA, clkB, rst, pulse_in;
wire pulse_out;

toggle_synchronizer dut (
    .clkA(clkA), .clkB(clkB), .rst(rst),
    .pulse_in(pulse_in), .pulse_out(pulse_out)
);

always #5  clkA = ~clkA;  // 10ns
always #7  clkB = ~clkB;  // 14ns (async)

initial begin
    clkA=0; clkB=0; rst=1; pulse_in=0;
    #20; rst=0;
    #15 pulse_in=1; #10 pulse_in=0;
    #40 pulse_in=1; #10 pulse_in=0;
    #30 pulse_in=1; #10 pulse_in=0;
    #100 $finish;
end
endmodule
04

Req/Ack Handshake Synchronizer

GUARANTEED DELIVERY

// Block Diagram

// Use Case

  • Reliable transfer of data between clock domains
  • Situations where event or data loss is unacceptable
  • Low-bandwidth, high-reliability control transactions

// Key Insight

Sender asserts REQ and holds data stable. Receiver captures data, asserts ACK. Sender sees ACK (synchronized back) and de-asserts REQ. Receiver sees REQ fall, de-asserts ACK. 4-phase handshake — both domains confirm each transition. No data is ever lost.

// Pros

  • Guaranteed delivery — zero data loss
  • Works regardless of clock frequency ratio
  • Can transfer multi-bit data safely (data stable during transfer)
  • Self-throttling — sender can't overrun receiver

// Cons

  • High latency: 4+ clock cycles per transfer (REQ sync + ACK sync)
  • Low throughput — only one transfer per full round-trip
  • Complex FSM logic at both sender and receiver
  • Deadlock possible if reset is not handled correctly

// Verification Checklist

  • Verify data bus is held stable from REQ assertion until ACK is received
  • Check FSM never deadlocks: timeout or watchdog recommended in silicon
  • Simulate with extreme clock ratios (10:1 and 1:10)
  • Verify reset sequence: both sides must recover to known idle state
  • Confirm back-to-back transfers work: second REQ must not begin until first ACK completes
  • CDC lint: REQ and ACK paths must each pass through a 2FF synchronizer
handshake_cdc_top.v + sender/receiver domains 
// ── TOP ──
module handshake_cdc_top #(parameter WIDTH=8) (
    input  wire             clk_a, clk_b, rstn, send,
    input  wire [7:0]       data_in,
    output wire [7:0]       data_out
);
wire req, ack, req_sync, ack_sync;
wire [7:0] data_reg;

sender_domain   sender  (.clk_a(clk_a), .rstn(rstn), .send(send),
    .data_in(data_in), .ack_sync(ack_sync), .req(req), .data_reg(data_reg));
sync_2ff        req_s   (.clk(clk_b), .rstn(rstn), .din(req),     .dout(req_sync));
receiver_domain receiver(.clk_b(clk_b), .rstn(rstn), .req_sync(req_sync),
    .data_in(data_reg), .ack(ack), .data_out(data_out));
sync_2ff        ack_s   (.clk(clk_a), .rstn(rstn), .din(ack),     .dout(ack_sync));
endmodule

// ── SENDER ──
module sender_domain (
    input  wire       clk_a, rstn, send, ack_sync,
    input  wire [7:0] data_in,
    output reg        req,
    output reg  [7:0] data_reg
);
always @(posedge clk_a or negedge rstn) begin
    if(!rstn) {req,data_reg} <= 0;
    else if(send && !req) begin data_reg<=data_in; req<=1; end
    else if(ack_sync)          req <= 0;
end
endmodule

// ── RECEIVER ──
module receiver_domain (
    input  wire       clk_b, rstn, req_sync,
    input  wire [7:0] data_in,
    output reg        ack,
    output reg  [7:0] data_out
);
always @(posedge clk_b or negedge rstn) begin
    if(!rstn) {ack,data_out} <= 0;
    else if(req_sync && !ack) begin data_out<=data_in; ack<=1; end
    else if(!req_sync)           ack <= 0;
end
endmodule
`timescale 1ns/1ps
module tb_handshake_cdc;
reg clk_a, clk_b, rstn, send;
reg  [7:0] data_in;
wire [7:0] data_out;

handshake_cdc_top dut (
    .clk_a(clk_a), .clk_b(clk_b), .rstn(rstn),
    .send(send), .data_in(data_in), .data_out(data_out)
);

initial clk_a=0; always #5  clk_a=~clk_a;
initial clk_b=0; always #7  clk_b=~clk_b;

initial begin
    rstn=0; send=0; data_in=0; #20; rstn=1;
    #20; data_in=8'hA5; send=1; #10; send=0;
    #300; data_in=8'h3C; send=1; #10; send=0;
    #300; $finish;
end
endmodule

Handshake-Based Pulse Synchronizer

EXACT ONCE

// Block Diagram

// Use Case

  • Safe transfer of single pulses with guaranteed delivery
  • When clocks have large frequency differences
  • Interrupt or event signaling across domains with confirmation

// Key Insight

A handshake mechanism converts the pulse into a controlled req–ack transaction. The source latches the event as REQ, which propagates to the destination. The destination generates pulse_dst and sends ACK back. Source clears REQ only after ACK is received — the event is captured exactly once.

// Pros

  • Guaranteed single pulse delivery — no duplication, no loss
  • Handles arbitrarily large frequency differences
  • Source knows when transfer is complete via busy signal
  • pulse_dst is precisely one destination clock cycle wide

// Cons

  • Cannot accept new events while busy (source must check busy)
  • Highest latency among pulse synchronizers
  • More logic than toggle synchronizer for same purpose
  • Careful reset required to avoid false ACK at startup

// Verification Checklist

  • Verify event_src is ignored when busy=1 (no missed updates)
  • Confirm pulse_dst is exactly 1 clkB cycle wide using edge detection
  • Check REQ deasserts only after ack_sync rises (synchronized ACK received)
  • Test with back-to-back events arriving just after busy deasserts
  • Simulate with clkB much faster AND much slower than clkA
  • Verify no glitch on busy signal during reset release
handshake_cdc.v — pulse version 
// ── SOURCE FSM ──
module cdc_source (
    input  wire clkA, reset, event_src, ack_sync,
    output reg  req,
    output wire busy
);
assign busy = req | ack_sync;
always @(posedge clkA or posedge reset) begin
    if(reset)             req <= 0;
    else if(event_src && !busy) req <= 1;
    else if(ack_sync)    req <= 0;
end
endmodule

// ── DESTINATION FSM ──
module cdc_dest (
    input  wire clkB, reset, req_sync,
    output reg  ack,
    output wire pulse_dst
);
reg req_sync_d;
always @(posedge clkB or posedge reset) begin
    if(reset) {req_sync_d, ack} <= 0;
    else begin
        req_sync_d <= req_sync;
        ack        <= req_sync;
    end
end
assign pulse_dst = req_sync & ~req_sync_d; // rising edge detect
endmodule

// ── TOP ──
module handshake_cdc (
    input  wire clkA, clkB, reset, event_src,
    output wire pulse_dst, busy
);
wire req, req_sync, ack, ack_sync;
cdc_source SRC (.clkA(clkA),.reset(reset),.event_src(event_src),
    .ack_sync(ack_sync),.req(req),.busy(busy));
sync_2ff   REQ (.clk(clkB),.reset(reset),.d(req),.q(req_sync));
cdc_dest   DST (.clkB(clkB),.reset(reset),.req_sync(req_sync),
    .ack(ack),.pulse_dst(pulse_dst));
sync_2ff   ACK (.clk(clkA),.reset(reset),.d(ack),.q(ack_sync));
endmodule
`timescale 1ns/1ps
module tb_handshake;
reg  clkA, clkB, reset, event_src;
wire pulse_dst, busy;

handshake_cdc DUT (
    .clkA(clkA), .clkB(clkB), .reset(reset),
    .event_src(event_src), .pulse_dst(pulse_dst), .busy(busy)
);

always #5  clkA = ~clkA;  // 100 MHz
always #12 clkB = ~clkB;  // different clock

initial begin
    clkA=0; clkB=0; reset=1; event_src=0;
    #20 reset=0;
    #17 event_src=1; #10 event_src=0;
    #100 event_src=1; #10 event_src=0;
    #200; $finish;
end
endmodule

MUX Synchronizer

MULTI-BIT DATA

// Block Diagram

// Use Case

  • Transferring multi-bit data between clock domains
  • When data changes infrequently and can be held stable
  • Configuration registers written from one domain, read by another

// Key Insight

Instead of synchronizing every data bit (which causes skew and corruption), a synchronized enable signal controls a MUX that decides when the destination register captures the data bus. The data itself is never synchronized — it just needs to be stable. The enable chain (FA → FB0 → FB1) ensures the destination samples only when data is guaranteed stable.

// Pros

  • Safe multi-bit data transfer without bus skew issues
  • Lower area than async FIFO for infrequent data updates
  • Simple structure — only enable signal crosses domain through sync chain
  • Parameterizable for any data width

// Cons

  • Data MUST remain stable for duration of enable propagation (~3 clkB cycles)
  • Not suitable for streaming or high-throughput data
  • Source domain must hold data — cannot pipeline transfers
  • Incorrect data hold time = silent data corruption (hard to debug)

// Verification Checklist

  • Verify data_in remains stable for ≥ 3 clkB cycles after EN asserts
  • Confirm data_out only changes on the correct clkB edge after enable propagation
  • Check MUX select logic: fb1 selects data_in (capture), 0 holds data_out (hold)
  • Simulate with rapidly changing data_in to expose hold-time violations
  • Verify multiple back-to-back transfers work with proper EN spacing
  • CDC lint: flag the data bus as intentional false path (data stable by design)
  • Run with setup/hold corner scenarios to verify no glitch at MUX output
mux_synchronizer.v / tb_mux_synchronizer.v 
module mux_synchronizer #(parameter WIDTH = 8) (
    input  wire             clka, clkb, en,
    input  wire [WIDTH-1:0] data_in,
    output reg  [WIDTH-1:0] data_out
);
reg fa, fb0, fb1;
wire [WIDTH-1:0] d1;

// CLKA domain: register enable
always @(posedge clka) fa <= en;

// CLKB domain: synchronize enable
always @(posedge clkb) begin
    fb0 <= fa;
    fb1 <= fb0;
end

// MUX: fb1=1 → capture data_in, fb1=0 → hold data_out
assign d1 = fb1 ? data_in : data_out;

// Destination register
always @(posedge clkb) data_out <= d1;

endmodule
`timescale 1ns/1ps
module tb_mux_synchronizer;
parameter WIDTH = 8;
reg  clka, clkb, en;
reg  [WIDTH-1:0] data_in;
wire [WIDTH-1:0] data_out;

mux_synchronizer #(.WIDTH(WIDTH)) dut (
    .clka(clka), .clkb(clkb),
    .en(en), .data_in(data_in), .data_out(data_out)
);

initial begin clka=0; forever #5 clka=~clka; end
initial begin clkb=0; forever #7 clkb=~clkb; end

initial begin
    en=0; data_in=0; #20;
    @(posedge clka); data_in=8'hA5; en=1;
    repeat(5) @(posedge clka); en=0; #100;
    @(posedge clka); data_in=8'h3C; en=1;
    repeat(5) @(posedge clka); en=0; #200;
end
endmodule

⚠ Classic Interview Trap

Never synchronize multi-bit buses directly using flip-flop chains.

If you put a multi-bit bus through a 2FF synchronizer, each bit resolves its metastability independently and at a different time. The result is a word with bits from two different clock cycles — functionally incorrect data with no X in simulation. This is one of the most common CDC bugs in real silicon. Use Gray code counters, async FIFOs, or the MUX synchronizer for multi-bit transfers.

CDC Method Comparison

Choose the right technique based on signal type, latency budget, and reliability requirement.

Method Signal Type Latency Throughput Multi-bit? Best For
2FF Sync Level 2 clkB cycles High No Enable, flag, reset
3FF Sync Level 3 clkB cycles High No High-freq, high reliability
Toggle Sync Pulse / Event ~4 clkB cycles Medium No Narrow pulses, any ratio
Req/Ack HS Data + Event High (4-phase) Low Yes Reliable data transfer
Pulse HS Pulse Highest Lowest No Exact-once pulse delivery
MUX Sync Multi-bit Data 3+ clkB cycles Low Yes Config regs, stable data
